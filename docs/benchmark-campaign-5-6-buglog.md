## Bug Log

### BENCH-001: Vast.ai `no_such_ask` not classified as retryable stale inventory error
- **Severity**: Critical
- **Provider**: Vast.ai
- **Sessions affected**: 13+ across campaigns 5 & 6
- **Status**: Open

**Symptom**: Vast.ai returns HTTP 400 with `no_such_ask` when an offer ID is no longer available:
```json
{"success":false,"error":"invalid_args","msg":"error 404/3603: no_such_ask  Instance type by id 29907796 is not available.","ask_id":29907796}
```

**Root cause**: The Vast.ai provider's `handleError()` maps HTTP 400 to `ErrProviderError` (generic). It does NOT check the response body for `no_such_ask`. This means:
1. `IsStaleInventoryError()` returns false
2. `ShouldRetryWithDifferentOffer()` returns false
3. The provisioner's auto_retry never fires for this class of error
4. The offer health tracker records it as `failureType: "unknown"`

**Contrast**: Blue Lobster correctly maps HTTP 409 `no_capacity` to `ErrOfferUnavailable`, which triggers auto_retry with a different offer.

**Fix**: In `internal/provider/vastai/client.go`, parse the HTTP 400 response body for `no_such_ask` and return `ErrOfferStaleInventory` or `ErrOfferUnavailable` instead of `ErrProviderError`.

**Files**: `internal/provider/vastai/client.go` (~line 1047)

### BENCH-002: Inventory cache TTL (1 min) defeats runner retry (10s gap)
- **Severity**: High
- **Provider**: All (most impactful on Vast.ai)
- **Sessions affected**: All retried entries using same stale offer
- **Status**: Open

**Symptom**: When a benchmark entry fails on attempt 1, the runner waits 10 seconds and retries. But `ListOffers()` has a 1-minute cache TTL (`DefaultCacheTTL = 1 * time.Minute`). The retry gets the exact same stale offer from cache, guaranteeing the same failure.

**Root cause**: The runner's retry delay (10s) is shorter than the inventory cache TTL (60s). No mechanism exists to invalidate a specific offer from cache after a provisioning failure.

**Fix options**:
1. Invalidate the failed offer from cache on `no_such_ask` / `no_capacity` errors
2. Pass `excludeOfferIDs` to `ListOffers()` on retry (the session already tracks `failed_offers`)
3. Increase retry delay beyond cache TTL (breaks responsiveness)

**Files**: `internal/service/inventory/service.go` (line 18), `internal/service/benchmark/runner.go` (line 411)

### BENCH-003: Vast.ai SSH `auth_failed` on provisioned instances
- **Severity**: High
- **Provider**: Vast.ai
- **Sessions affected**: 9 in campaign 6, ~5 in campaign 5
- **Status**: Open

**Symptom**: Vast.ai successfully provisions an instance (receives IP + port), but SSH verification fails with `permanent SSH error: auth_failed`. The SSH key generated by the provisioner is rejected by the Vast.ai host. After 3 consecutive `auth_failed` responses, the session is marked as permanently failed.

**Observed GPUs**: RTX 5060 Ti, RTX 5080, RTX 4090 (not RTX 5070 Ti)

**Root cause**: Unknown — likely host-side. Possible causes:
- Host hasn't finished initializing SSH authorized_keys when we connect
- Key format incompatibility with certain host configurations
- Race condition between Vast.ai instance creation and SSH key injection

**Current behavior**: `auth_failed` is treated as a permanent SSH error. The provisioner destroys the instance and fails the session. Auto_retry does NOT trigger with a different offer for SSH failures.

**Fix**: Consider treating `auth_failed` as retryable with a different offer, since it's likely host-specific. Or add a longer delay before first SSH attempt to give the host more time.

**Files**: `internal/service/provisioner/service.go` (~lines 1053-1067)

### BENCH-004: Vast.ai HTTP 429 rate limiting during parallel benchmark dispatch
- **Severity**: Medium
- **Provider**: Vast.ai
- **Sessions affected**: 2 in campaign 6
- **Status**: Open

**Symptom**: Vast.ai returns HTTP 429 during parallel provisioning:
```json
{"message": "The action could not be performed because there were too many requests by the client.\n\nAPI requests too frequent: endpoint threshold=4.5*1.0\n\n", "code": "429 Too Many Requests"}
```

**Root cause**: The benchmark runner dispatches all 12 entries with only 2-second stagger. Each entry's provisioner also makes inventory + create calls. This exceeds Vast.ai's rate limit (`threshold=4.5*1.0` = 4.5 requests per second).

**Current behavior**: HTTP 429 is not retried — it's mapped to `ErrProviderError` and the provisioning attempt fails immediately.

**Fix options**:
1. Map HTTP 429 to a retryable error with exponential backoff
2. Increase dispatch stagger for Vast.ai entries (e.g., 5s instead of 2s)
3. Add request-level rate limiting to the Vast.ai client (currently only BL has this)

**Files**: `internal/provider/vastai/client.go`, `internal/service/benchmark/runner.go`

### BENCH-005: Blue Lobster RTX 5090 capacity exhaustion with concurrent entries
- **Severity**: Medium
- **Provider**: Blue Lobster
- **Sessions affected**: 3 in each campaign (6 total)
- **Status**: Open (provider-side limitation)

**Symptom**: Blue Lobster has only 1 RTX 5090 instance available. When the first entry claims it, subsequent entries for the same GPU fail with:
```
HTTP 409: no_capacity: No capacity available in igl for v2_gpu_1x_5090
```

**Root cause**: Limited supply (1 slot) + concurrent dispatch. The first qwen2:7b entry succeeds, but llama and deepseek entries fail because the slot is occupied for ~10 min.

**Behavior**: BL correctly returns 409 → `ErrOfferUnavailable` → auto_retry fires, but there are no alternative offers. Both runner-level retries fail within the 10s gap (slot still occupied by first entry).

**Fix**: Serialize entries for the same GPU type on single-capacity providers. Run models sequentially per GPU instead of in parallel.

### BENCH-006: Blue Lobster RTX 8000 task poll timeout (5 min) under concurrent load
- **Severity**: Medium
- **Provider**: Blue Lobster
- **Sessions affected**: 1-2 per campaign
- **Status**: Open

**Symptom**: BL creates the VM successfully (vm_uuid assigned) but the instance fails to reach ready state within the 5-minute poll timeout. Happens under concurrent load when multiple benchmark entries provision simultaneously.

**Root cause**: BL's backend takes longer to provision when multiple VMs are launching. The RTX 8000 is particularly slow (also affected by 15-30 min driver stabilization).

**Fix**: Increase poll timeout from 5 min to 10 min, or implement a per-GPU-type timeout configuration.

**Files**: `internal/provider/bluelobster/client.go`

### BENCH-007: Blue Lobster RTX A4000 disappeared from inventory
- **Severity**: Low
- **Provider**: Blue Lobster
- **Sessions affected**: 3 in campaign 6 (all A4000 entries)
- **Status**: Informational

**Symptom**: RTX A4000 was available in campaign 5 ($0.20/hr) but gone from BL inventory in campaign 6. All 3 A4000 entries failed immediately — the runner couldn't find any offers.

**Root cause**: Provider-side capacity fluctuation. A4000 availability is not guaranteed.

**Runner behavior**: When no offers exist for a GPU type, the runner still burns both retry attempts checking inventory. The failure reason is generic "all 2 attempts failed" with no clarity that the GPU type simply has zero offers.

**Fix**: Log a clear message when zero offers found for a GPU type and fail immediately instead of retrying. Example: "no offers available for RTX A4000 on bluelobster".

### BENCH-009: Vast.ai orphaned instances from failed sessions not cleaned up at startup
- **Severity**: Medium
- **Provider**: Vast.ai
- **Sessions affected**: 13 orphaned instances found on server restart
- **Status**: Open

**Symptom**: When the server restarted for campaign 6, it found 13 Vast.ai instances still running from campaign 5's failed sessions:
```
found failed session with provider instance session_id=03e27e15 provider_id=31988337
```

These are instances where provisioning created a Vast.ai VM, but the session was marked failed (e.g., due to SSH auth_failed). The instance was never destroyed.

**Root cause**: When `auth_failed` or other SSH errors fail a session, the code calls `failSession()` which updates status to failed, but the destroy path may not execute if the failure happens during the verification loop. The startup sweep detects these but the current run's logs show they were found but the cleanup action is unclear.

**Fix**: Ensure `failSession()` always triggers `DestroyInstance()` for the associated provider instance. The startup sweep should also destroy instances for failed sessions, not just warn about them.

**Files**: `internal/service/provisioner/service.go`
